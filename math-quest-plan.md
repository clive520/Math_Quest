# 數學闖關網站設計計畫書

## 一、網站願景

本網站希望成為一個給國小老師與學生使用的數學闖關平台。學生可以在遊戲化的關卡中複習數學，老師則可以建立班級、管理學生、設計題目、指派任務，並查看學生的作答狀況。

除了班級內使用之外，本網站也希望具有「老師共備」與「題庫共享」的精神。每位老師設計的題目都可以公開呈現在網站上，其他註冊老師可以直接複製成自己的題目，再依照班級程度修改後使用。

核心目標：

- 讓學生覺得自己是在闖關，而不是單純寫練習卷。
- 讓老師能快速出題、派題、追蹤學習狀況。
- 讓老師彼此分享題目，減少重複備課時間。
- 透過動態數字設計，降低學生互相抄答案的可能。

## 二、使用者角色

### 1. 老師

老師是主要管理者，可以：

- 註冊與登入帳號。
- 建立一個或多個班級。
- 建立學生帳號。
- 建立數學題目。
- 將題目整理成任務或關卡。
- 指派題目給指定班級或學生。
- 查看學生作答結果與闖關進度。
- 將自己的題目公開分享給其他老師。
- 複製其他老師公開的題目，修改後成為自己的題目。

### 2. 學生

學生可以：

- 使用老師建立的帳號登入。
- 查看老師指派的任務或關卡。
- 進行數學闖關。
- 查看自己的分數、星星、完成狀態與進度。
- 重練錯題或重新挑戰任務。

### 3. 系統管理者

系統管理者初期可以先簡化，未來再加入：

- 管理不適合公開的題目。
- 管理違規帳號。
- 查看平台整體使用狀況。
- 維護公開題庫分類。

## 三、帳號與班級管理

### 老師帳號

老師可以自行註冊帳號。建議第一版支援：

- Email + 密碼註冊。
- Google 帳號登入，若技術上容易整合可加入。
- 忘記密碼功能。

每位老師登入後，只能管理自己建立的班級、學生、派題任務與私人題目。

### 班級管理

老師可以建立多個班級，例如：

- 三年甲班
- 四年乙班
- 數學補救小組

班級資料可包含：

- 班級名稱
- 年級
- 學期
- 建立老師
- 班級代碼
- 是否封存

### 學生管理

老師可以在班級中建立學生資料。

學生資料建議包含：

- 姓名
- 座號
- 班級
- 登入帳號或登入代碼
- 初始密碼
- 闖關進度
- 最近登入時間

學生登入方式建議第一版採用：

> 班級代碼 + 座號或學生代碼 + 密碼

這比要求每位國小學生使用 Email 更適合教室現場。

## 四、老師出題系統

老師可以建立題目，題目分成「固定題」與「動態題」。

### 固定題

固定題是每位學生看到完全相同的題目。

範例：

```text
25 + 37 = ?
```

### 動態題

動態題是本網站的核心功能。老師設定題型與數字範圍，系統在學生作答時自動產生不同數字。

範例：

```text
{{a}} + {{b}} = ?
```

變數設定：

```json
{
  "a": { "min": 10, "max": 99 },
  "b": { "min": 10, "max": 99 }
}
```

答案規則：

```text
a + b
```

不同學生看到的題目可能是：

- 學生 A：34 + 58 = ?
- 學生 B：72 + 19 = ?
- 學生 C：46 + 37 = ?

這樣題型一致，方便老師檢查學習目標；數字不同，能降低學生直接互相抄答案的情況。

### 題目類型

第一版建議支援：

- 填空題
- 選擇題
- 是非題
- 四則運算動態題

後續可加入：

- 分數題
- 小數題
- 單位換算
- 應用題
- 幾何題
- 多步驟解題

## 五、老師派題系統

老師可以把題目組成「任務」或「關卡」，再指派給班級。

派題設定可包含：

- 任務名稱
- 任務說明
- 指派班級
- 題目清單
- 開放時間
- 截止時間
- 是否允許重複挑戰
- 是否顯示正確答案
- 是否顯示詳解
- 過關條件

過關條件範例：

- 答對率達 80%。
- 全部答對才過關。
- 完成即可過關。
- 三次挑戰內達成指定分數。

學生登入後，只會看到老師指派給自己班級或自己的任務。

## 六、題庫共享與複製機制

本網站希望成為一個「無私」的教學共備平台，因此題目共享是重要功能。

### 題目公開設定

老師建立題目時，可以選擇：

- 私人題目：只有自己看得到。
- 公開題目：其他老師可以搜尋、查看、複製。
- 暫不公開：先自己使用，之後再公開。

### 公開題庫頁面

公開題庫可以讓老師用以下條件搜尋：

- 年級
- 單元
- 題型
- 關鍵字
- 建立老師
- 使用次數
- 最新上架
- 熱門題目

### 複製使用

其他老師看到公開題目後，可以按「複製到我的題庫」。

複製後：

- 原題目不會被修改。
- 複製者會得到一份自己的版本。
- 複製者可以自由調整數字範圍、題目文字、答案規則與難度。
- 系統可保留原作者資訊，作為感謝與來源紀錄。

### 題目授權精神

網站可以在規則中說明：

> 老師公開題目後，代表同意其他註冊老師於教學用途上複製、修改與使用。

這樣能讓平台的共享精神更清楚，也避免日後產生誤會。

## 七、學生闖關體驗

學生登入後，看到的第一畫面應該是自己的任務與關卡，而不是複雜選單。

建議畫面包含：

- 今日任務
- 尚未完成的關卡
- 已完成的關卡
- 星星數
- 個人進度
- 老師給的鼓勵訊息

遊戲化元素建議第一版先做簡單、穩定、對學習有幫助的設計：

- 關卡卡片
- 星星評分
- 完成勾選
- 答對動畫
- 錯題重練
- 個人進度條

第二階段再考慮：

- 班級共同進度
- 徽章
- 頭像
- 地圖式關卡
- 團隊任務

排行榜要謹慎使用，避免造成低成就學生挫折。若要做，建議使用「進步榜」或「完成率」取代單純速度排名。

## 八、資料架構初步規劃

使用 Vercel + Supabase 時，資料庫會使用 Supabase 的 PostgreSQL。建議初步資料表如下：

### teachers

儲存老師資料。

欄位範例：

- id
- email
- display_name
- created_at

### classes

儲存班級資料。

欄位範例：

- id
- teacher_id
- name
- grade
- semester
- class_code
- archived
- created_at

### students

儲存學生資料。

欄位範例：

- id
- class_id
- name
- seat_number
- login_code
- password_hash 或 auth_user_id
- created_at

### question_templates

儲存題目模板。

欄位範例：

- id
- owner_teacher_id
- title
- subject
- grade
- unit
- question_type
- template
- variables
- answer_rule
- explanation
- visibility
- source_question_id
- created_at
- updated_at

visibility 可分為：

- private
- public
- unlisted

source_question_id 用來記錄這題是否從其他老師的公開題目複製而來。

### assignments

儲存老師派發的任務。

欄位範例：

- id
- teacher_id
- class_id
- title
- description
- start_at
- end_at
- pass_rule
- allow_retry
- created_at

### assignment_questions

儲存任務中包含哪些題目。

欄位範例：

- id
- assignment_id
- question_template_id
- order_index
- points

### student_attempts

儲存學生每次作答紀錄。

欄位範例：

- id
- student_id
- assignment_id
- question_template_id
- generated_values
- rendered_question
- student_answer
- correct_answer
- is_correct
- answered_at

generated_values 很重要，用來記錄該學生當次拿到的動態數字。例如：

```json
{
  "a": 34,
  "b": 58
}
```

這樣學生重新整理頁面時，題目不會一直變動，也方便老師事後檢查。

## 九、技術選型

本專案決定使用：

- 前端與網站部署：Vercel
- 後端資料庫與登入：Supabase
- 程式碼管理：GitHub

### Vercel 的角色

Vercel 負責：

- 部署網站
- 自動從 GitHub 更新網站
- 提供正式網址
- 管理環境變數
- 支援 Next.js 專案

### Supabase 的角色

Supabase 負責：

- 老師登入
- 資料庫
- 題庫資料
- 班級與學生資料
- 作答紀錄
- 權限控管

Supabase 使用 PostgreSQL，日後做報表、統計、匯出資料會比較清楚。

### GitHub 的角色

GitHub 負責：

- 保存程式碼
- 記錄修改歷史
- 與 Vercel 串接，自動部署

GitHub 不建議作為正式網站主機，因為本網站需要帳號、班級、學生資料與資料庫。GitHub Pages 較適合靜態網站，不適合這類完整應用系統。

## 十、自動化開發與部署目標

因為希望操作越少越好，專案會盡量採用自動化流程。

### 預期自動化項目

- 程式碼推送到 GitHub 後，自動部署到 Vercel。
- Vercel 自動產生預覽網站。
- 正式分支更新後，自動更新正式網站。
- Supabase 資料庫結構用 migration 檔管理。
- 本地開發環境盡量用固定指令啟動。
- 環境變數集中管理。
- 未來可加入自動測試，避免更新後破壞登入、派題、作答等核心功能。

### 老師需要操作的部分

目標是讓使用者，也就是老師，只需要在網站中操作：

- 註冊
- 建班
- 建學生
- 出題
- 派題
- 查看結果

技術面的部署、資料庫更新、版本管理，盡量由開發流程自動處理。

## 十一、開發階段規劃

### 第一階段：平台基礎版

目標是讓多位老師可以註冊並各自使用。

功能包含：

- 老師註冊與登入
- 老師建立班級
- 老師建立學生
- 學生登入
- 老師建立私人題目
- 老師建立動態題目
- 老師派題給班級
- 學生作答
- 老師查看作答結果

### 第二階段：公開題庫共享

功能包含：

- 題目公開設定
- 公開題庫瀏覽
- 題目搜尋與篩選
- 複製公開題目到我的題庫
- 保留原作者來源
- 題目使用次數統計

### 第三階段：闖關遊戲化

功能包含：

- 關卡地圖
- 星星評分
- 錯題重練
- 徽章
- 班級共同進度
- 個人化頭像

### 第四階段：教學分析

功能包含：

- 每題答對率
- 每位學生弱點分析
- 班級常錯題型
- 匯出 Excel
- 根據錯題自動建立補救任務

## 十二、安全與隱私原則

因為網站會保存學生資料，必須從第一版就注意安全。

基本原則：

- 不保存學生身分證字號。
- 不保存學生生日、電話、地址等非必要個資。
- 學生登入採用代碼即可，不強迫使用 Email。
- 老師只能看到自己的班級與學生。
- 公開題庫只公開題目，不公開學生作答資料。
- 資料庫必須設定 Row Level Security，避免不同老師互相讀取私人資料。
- 密碼不可用明文儲存。

## 十四、建議的第一版範圍

為了讓網站能盡快實際使用，第一版建議先聚焦：

1. 多位老師註冊。
2. 老師各自建立班級。
3. 老師建立學生登入資料。
4. 老師建立動態四則運算題。
5. 老師指派題目給班級。
6. 學生登入後完成作答。
7. 老師查看學生答題結果。

公開題庫可以先設計資料結構，畫面與完整搜尋功能放到第二階段。

## 十五、目前決策

目前已確認：

- 網站定位為多位老師可註冊使用的平台。
- 題目可公開分享給其他老師。
- 其他老師可以複製公開題目，修改後自己使用。
- 技術選型採用 Vercel + Supabase。
- 希望整個開發、部署與資料庫管理流程盡量自動化，減少手動操作。
